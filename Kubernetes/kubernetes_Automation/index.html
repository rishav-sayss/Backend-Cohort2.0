An AI-powered cloud
sandbox IDE

codeSpace provisions isolated React + Vite sandboxes inside Kubernetes on demand. Each sandbox ships with a live preview, an integrated terminal, and a Mistral-powered AI agent that reads and writes files directly in the running pod — hot-reloading instantly through Vite's HMR.

01
Executive summary
What it is, who it's for, and what makes it different

codeSpace is a browser-based cloud IDE that spins up isolated, per-user React + Vite environments on demand inside a Kubernetes cluster. Each sandbox runs as a dedicated pod pre-loaded with a starter template. Users interact with a VS Code-style UI that includes a live preview pane, a file explorer, an integrated terminal, and an AI chat panel powered by Mistral Large via LangChain/LangGraph.

The AI agent reads, creates, and updates files directly inside the running pod in real time, triggering Vite's HMR so changes appear instantly in the preview without a full reload.

Core value proposition

Spin up a production-quality React sandbox in one click, describe what you want to build in plain English, and watch the AI write and hot-reload the code while you observe it live in the same browser tab.

Target users

  • Frontend developers who want a zero-setup coding environment
  • Students & learners exploring React without local toolchain pain
  • Educators running classes where each student needs a working sandbox in seconds

Per-sandbox pod (created on demand)

Every active session has its own pod with two containers sharing an emptyDir volume mounted at /workspace:

  • template container — runs npm run dev, exposes Vite on port 5173
  • agent container — file API + PTY over Socket.IO on port 3000

An init container seeds the shared volume from the template image's /workspace. Both runtime containers see the same file tree, so writes from the agent are picked up by Vite's watcher.

Why a separate router

nginx ingress can't read the dynamic subdomain ID and route to a per-sandbox ClusterIP service without one rule per sandbox. The Router parses req.headers.host at request time and proxies to the right backend — a single ingress entry handles all sandboxes.

Sandbox lifecycle
From "Create Sandbox" click to TTL-driven cleanup

Provisioning flow

User clicks "Create Sandbox" on SplashScreen
Frontend issues POST /api/sandbox/start
Ingress routes to sandbox-service (ClusterIP)
/api/sandbox prefix matches the sandbox deployment
Sandbox server generates UUIDv7 → sandboxId
Used as pod name suffix, service selector, and Redis key
Pod + service + Redis key created in parallel
createNamespacedPod, createNamespacedService, SET sandbox:{id} EX 120
Response returned to client
{ sandboxId, previewUrl: "http://{id}.preview.localhost" }

Pod specification

Init container

Image template. Runs sh -c 'cp -r /workspace/. /seed/' — copies the pre-built React/Vite template from the image into the shared emptyDir volume.

Container 1 — sandbox-container

Image template. Starts Vite dev server. Port 5173. CPU 250m/500m, memory 500Mi/1Gi.

Container 2 — agent-container

Image agent. File API + PTY. Port 3000. CPU 250m/500m, memory 500Mi/1Gi.

Why shared emptyDir

Both containers mount workspace-volume at /workspace. The init container seeds it once. After that, every file the agent writes is immediately visible to Vite's file watcher — which is exactly what enables instant HMR after AI edits.

Per-sandbox K8s service

sandbox-service-{id}.yaml (generated)
ports:
  - name: http        port: 80   targetPort: 5173  # Vite preview
  - name: agent-http  port: 3000 targetPort: 3000  # Agent API
selector:
  sandboxId: {sandboxId}   # unique — prevents cross-sandbox routing
type: ClusterIP
💡

The selector uses sandboxId, not a shared label like app: sandbox. This guarantees each per-sandbox service routes only to its paired pod — no cross-tenant traffic possible.

TTL & cleanup policy

  • On creation, sandbox:{id} is stored in Redis with a 120-second TTL.
  • Every HTTP / WebSocket request through the Router calls EXPIRE sandbox:{id} 120 — TTL is effectively activity-based.
  • An ioredis subscriber in the sandbox server listens on __keyevent@0__:expired. On expiry it calls deleteNamespacedPod + deleteNamespacedService with gracePeriodSeconds: 0.
  • No persistent storage — the emptyDir volume dies with the pod.

Subdomain routing

The nginx ingress maps both wildcard hosts to the router:

ingress (wildcard rules)
*.preview.localhost  →  router-service :80
*.agent.localhost    →  router-service :80

The Router reads req.headers.host, splits on . to extract sandboxId and type (preview vs agent), and proxies to the matching sandbox-service-{sandboxId} ClusterIP.

05
AI code editing layer
How user intent becomes file writes inside a running pod

Agent architecture

The AI layer uses LangChain + LangGraph to build a stateful ReAct agent:

agent.js (simplified)
LangGraph.createAgent({
  model:  ChatMistralAI("mistral-large-latest"),
  tools:  [list_files, read_files, update_files],
  systemPrompt: "codeSpace — expert AI frontend engineer...",
  recursionLimit: 100
})

Standard tool-calling loop: the agent receives a user message, decides which tools to call, executes them, receives results, and repeats until it has enough information to produce a final response.

Streaming response (SSE)

The POST /api/ai/invoke endpoint sets Content-Type: text/event-stream and flushes text over the open connection:

Output typeSourceRendered as
Activity lineswriter callbackTool progress: "Listing files...", "Reading files...", "Updating files..."
Final responselast ai messageThe last ai-role message with no tool_calls — written, then res.end() closes the stream.

The frontend's AiChat reads the stream with the Fetch API's ReadableStream, classifies each line, and renders an expandable activity log under the AI bubble.

Tools the agent has

ToolHTTP callReturns
list_filesGET .../list-filesFlat array of all file paths in /workspace.
read_filesGET .../read-files?files=…Map of { path: content } for requested files.
update_filesPATCH .../update-filesWrites full file contents. Creates parent directories.

All tools receive config.context.projectId and config.context.writer via LangGraph's runnable config. The agent reaches the sandbox agent sidecar over Kubernetes internal DNS: http://sandbox-service-{sandboxId}:3000 resolves to the per-sandbox ClusterIP.

Agent workflow (from system prompt)

  1. Understand — parse user intent, identify implicit requirements.
  2. Plan — outline component tree, styling approach, sections.
  3. Explore — list_files → read_files on entry points and target files.
  4. Build — update_files in batched calls: configs first, shared components next, pages last.
  5. Polish — verify responsiveness, imports, accessibility.
  6. Report — summarize changes (written to stream, not to files).

Hot reload integration

File writes from update_files land on the shared emptyDir volume. Vite's file watcher (configured with usePolling: true, interval: 300ms) detects the change and sends an HMR update over the WebSocket at ws://{id}.preview.localhost. The PreviewFrame iframe receives the update and patches the DOM without a full reload.

Model parameters

ParameterValue
ProviderMistralAI
Model IDmistral-large-latest
Temperature0.7
Max recursion100 LangGraph steps

Context window management

The agent uses the full LangGraph message history within a single invocation. There is currently no cross-session memory — each POST /api/ai/invoke starts a fresh agent run with only the current user message. The agent always calls list_files first to orient itself, then read_files on relevant files before making edits.

Error handling

  • If the agent throws during streaming, the catch block checks res.headersSent. If headers were already sent, it calls res.end() gracefully; otherwise it returns 500 { error: "Failed to invoke agent" }.
  • Individual tool errors (file not found, etc.) return error strings in the tool result object rather than throwing — the agent can reason about the failure and retry or skip.
  • File writes include mkdir -p semantics (recursive: true) so the agent can create files in new directories without a separate step.

No PVCs. All sandbox storage uses emptyDir volumes — ephemeral, tied to pod lifetime. Intentional: sandboxes are throwaway. Add PVC-backed storage if save/resume is needed.

Autoscaling

💡

Security considerations
Where we're solid, where we need to harden

Sandbox isolation model

  • Pod-level isolation: each sandbox is its own pod; containers share a network namespace within the pod but are isolated from other pods.
  • No cross-sandbox routing: the per-sandbox service selector uses unique sandboxId, not a shared label.
  • No host mount: the emptyDir volume is ephemeral and kernel-managed; it can't escape the pod boundary.
  • Resource limits enforced: CPU and memory limits cap runaway consumption.
⚠️

No securityContext on sandbox containers — no readOnlyRootFilesystem, no runAsNonRoot, no allowPrivilegeEscalation: false. Users have arbitrary terminal access inside the sandbox. Acceptable for a controlled-access dev environment; must be hardened before public deployment.

Authentication & authorization

ConcernTodayTarget
User authGoogle OAuth + JWT cookieEnforce JWT on /api/sandbox/start and /api/ai/invoke
Service-to-servicePlain HTTP within clustermTLS via Istio or cert-manager
Agent APINo auth — open within clusterShared secret header validated by agent
K8s APIresource-manager role — pods + services onlyAlready correct ✓

Secret management — critical

⚠️

k8s/secrets.yml contains live credentials in plaintext and is committed to the repository. Remediation steps:

  1. Remove k8s/secrets.yml from git history using git filter-repo or BFG Repo Cleaner.
  2. Rotate every exposed credential: MongoDB, Redis, RabbitMQ, Google OAuth, JWT secret, MistralAI key.
  3. Replace with one of:
    • Bitnami Sealed Secrets — client-side encryption, ciphertext in git
    • External Secrets Operator — pulls from AWS Secrets Manager / Vault at deploy time
    • SOPS + age — encrypt the secrets file, decrypt in CI

Input validation

  • AI agent accepts arbitrary user prompts. No prompt injection guard or content filter at the API boundary.
  • File paths in the agent REST API are joined with path.join(WORKING_DIR, file) — prevents traversal above /workspace on POSIX.
  • No rate limiting on /api/ai/invoke or /api/sandbox/start.

Network policies

💡

No NetworkPolicy resources today. Recommended: allow sandbox pods to receive only from router-service and ai-service; deny all egress except kube-dns; isolate notification service to RabbitMQ inbound only.

Observability
Logs, metrics, traces, alerts, probes

Logging

All services use morgan HTTP request logging in dev format. Application logs go to stdout/stderr and are collected by the Kubernetes node log driver.

Gaps: no structured JSON logging, no correlation IDs, no centralized log aggregation. Recommended: Fluent Bit → OpenSearch → Kibana, or Loki + Grafana.

Metrics & monitoring

💡

No Prometheus metrics today. Add prom-client to each service, expose /metrics. Instrument HTTP request rate, latency histograms, active sandbox count, Redis TTL operations, agent invocation count + duration. Deploy via kube-prometheus-stack.

Tracing

No distributed tracing implemented. Suggested: OpenTelemetry SDK in each service, exporting to Jaeger or Tempo. Key traces to capture: sandbox creation end-to-end, AI agent tool-call chain.

Alerting

No alerting rules. Recommended Prometheus alerts:

  • Sandbox pod crash loop (kube_pod_container_status_restarts_total > 5 in 5m)
  • Redis connection errors in sandbox-server or router
  • AI invocation error rate > 10% in 5m
  • No healthy replicas for auth or ai-orchestration

Contributing guide
Branch names, PR process, code style, commit format

Branch naming

PatternUse
feat/<desc>New features
fix/<desc>Bug fixes
chore/<desc>Tooling, deps, config (no production code change)
docs/<desc>Documentation only
refactor/<desc>Code restructuring with no behavior change

Examples: feat/sandbox-persistence, fix/ai-env-var-mismatch

Pull request process

  1. Branch from main. Rebase (don't merge) before opening the PR.
  2. Title format: <type>(<scope>): <subject> — e.g. fix(ai): correct MISTRAL_API_KEY env var name
  3. PR description must include: What changed and why, link to related issue, how to test locally.
  4. All CI checks must pass.
  5. Require at least one reviewer approval before merge.

Review checklist

Before approving a PR

• No secrets or credentials in code or manifests
• No console.log left in production paths
• New endpoints have at least a basic integration test
• K8s manifests include resource requests/limits
• Health probes updated if new service or port added
• This documentation is updated if architecture changed

Code style

  • All services: ESLint with the project default config (eslint.config.js).
  • Frontend: React-specific rules — eslint-plugin-react-hooks, eslint-plugin-react-refresh.
  • All Node services use "type": "module" — use import/export, not require.
  • Run npm run lint in each changed service before committing.

Commit message convention

Conventional Commits:

Format
<type>(<scope>): <subject>

[optional body]

[optional footer: Co-Authored-By, Fixes #issue]

Types: feat, fix, chore, docs, refactor, test, perf.
